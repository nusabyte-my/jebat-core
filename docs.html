Local Development

Run JEBAT locally for development and testing.

# Clone the repository
git clone https://github.com/nusabyte-my/jebat-core.git
cd jebat-core

# Install dependencies
pip install -r requirements.txt

# Copy environment template
cp .env.example .env

# Start the API server
uvicorn main:app --host 0.0.0.0 --port 8080 --reload

# In another terminal, launch the REPL
jebat repl

The API server starts at http://localhost:8080. Available endpoints:

  • / — API root and liveness check
  • /health — Health probe
  • /swagger — Swagger UI (interactive API explorer)
  • /webui/ — Web console

VPS Production Deployment

Deploy to a private VPS with PM2 process management.

Server Setup

# SSH into your VPS
ssh root@your-vps-ip

# Install Node.js and Python
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt install -y python3.12 python3-pip nodejs

# Install PM2
npm install -g pm2

# Clone the project
git clone https://github.com/nusabyte-my/jebat-core.git /var/www/jebat-core
cd /var/www/jebat-core

# Install dependencies
pip install -r requirements.txt

# Configure environment
cp .env.example .env
nano .env  # Edit with production values

Start with PM2

# Start the API server
pm2 start "python -m uvicorn main:app --host 0.0.0.0 --port 8080 --workers 4" --name jebat-api

# Save process list
pm2 save

# Auto-start on boot
pm2 startup

# Monitor logs
pm2 logs jebat-api

Air-Gapped Deployment

Run JEBAT entirely offline with no external dependencies.

Using Ollama

# 1. Install Ollama on the air-gapped server
# (transfer the installer via USB or internal network)
ollama serve &

# 2. Pull models (on a connected machine first, then transfer)
ollama pull qwen2.5-coder:7b

# 3. Configure JEBAT to use local Ollama
JEBAT_LLM_PROVIDER=ollama
JEBAT_LLM_MODEL=qwen2.5-coder:7b
OLLAMA_HOST=http://localhost:11434

# 4. Launch JEBAT
jebat repl

Using llama.cpp with GGUF Weights

# 1. Install llama.cpp (transfer binary to air-gapped server)
llama-server --model model.gguf --port 8081

# 2. Configure JEBAT
JEBAT_LLM_PROVIDER=local
LLAMA_CPP_HOST=http://localhost:8081
💡 Tip: For production air-gapped deployments, use the .206 → .65 remote model host topology where a dedicated model server (72.62.255.206) hosts llama.cpp and the API server (72.62.254.65) routes to it over the internal network.

LLM Provider Guide

JEBAT routes inference requests through a chain of providers with automatic failover, cost tracking, and health monitoring.

How Failover Works

  1. JEBAT sends the request to the primary provider (e.g., ollama)
  2. If the provider fails (timeout, rate limit, auth error), JEBAT tries the next in the fallback chain
  3. The chain continues until a provider succeeds or all are exhausted
  4. The successful provider name is returned in the response metadata

OpenRouter — Multi-Provider Gateway

OpenRouter — Multi-Provider Gateway

OpenRouter provides a unified API gateway to 200+ models from OpenAI, Anthropic, Google, Meta, Mistral, and other providers through a single OPENROUTER_API_KEY. JEBAT treats OpenRouter as a first-class provider with automatic failover.

Configuration

# ~/.jebat/config.yaml
llm:
  provider: openrouter
  model: openai/gpt-4o
  fallbacks:
    - ollama
    - local

Environment Variable

export OPENROUTER_API_KEY="sk-or-v1-..."
export JEBAT_LLM_PROVIDER=openrouter
export JEBAT_LLM_MODEL=openai/gpt-4o

Key Features

  • Model catalog: Access 200+ models from multiple providers through a single endpoint
  • Cost routing: Automatically select the cheapest model that meets quality thresholds
  • Failover target: Ideal as a cloud fallback when local Ollama or llama.cpp is unavailable
  • No GPU required: All inference runs on OpenRouter's infrastructure

llama.cpp — Local GGUF Inference

llama.cpp enables fully air-gapped local inference using quantized GGUF model weights. JEBAT communicates with a llama.cpp server via its OpenAI-compatible API, making it a drop-in replacement for cloud providers in offline deployments.

Configuration

# ~/.jebat/config.yaml
llm:
  provider: local
  model: qwen2.5-coder-7b
  fallbacks:
    - ollama

Environment Variable

export LLAMA_CPP_HOST="http://127.0.0.1:8080"
export JEBAT_LLM_PROVIDER=local
export JEBAT_LLM_MODEL=qwen2.5-coder-7b

  • OpenAI-compatible API: Drop-in replacement for cloud providers with identical request format
  • Custom weights: Load any GGUF model including fine-tuned and community models

Memory System

JEBAT implements a 5-layer eternal memory system with heat-scored retrieval for persistent cross-session context.

Memory Layers

LayerPurposeStorageRetention
Sensory (M0)Live working contextIn-memoryCurrent session
Episodic (M1)Session logs & eventsPostgreSQL30 days
Semantic (M2)Concepts & factsPostgreSQLPermanent
Conceptual (M3)Abstracted patternsPostgreSQLPermanent
Procedural (M4)Action patterns & skillsPostgreSQLPermanent

Heat Scoring

Memories are scored by recency, access frequency, and relevance. Frequently accessed memories stay "hot" and are retrieved first. Unused memories cool down over time but are never deleted.

CLI Usage

# Store a memory
jebat memory store "API uses Bearer token auth"

# Search memories
jebat memory search "authentication"

# View memory stats
jebat memory stats

Agent Orchestration

JEBAT orchestrates 17 specialized agent roles through a ReAct (Reasoning and Acting) loop.

Agent Roles

RoleMalay NameSpecialization
BuilderTukangCode generation, file creation, implementation
ExecutorHulubalangCommand execution, deployment, operations

ReviewerPengacaraCode review, quality analysis, compliance

ResearcherPenyelidikWeb search, documentation, knowledge gathering
SecurityKeselamatanVulnerability scanning, hardening, pentesting

ReAct Loop

  1. Reason: The agent analyzes the task and decides the next action
  2. Act: The agent selects and executes a tool (file write, terminal command, etc.)
  3. Observe: The agent reads the tool output
  4. Iterate: Steps 1-3 repeat until the task is complete or max iterations reached
# Delegate a task to the swarm
jebat agent "Audit the authentication module for vulnerabilities"

# Run in planning mode
jebat agent --mode strategic "Design a microservice architecture"

Architecture

JEBAT is built on a layered architecture designed for sovereign, air-gap-compatible deployment.

System Layers

LayerComponentsTech
API GatewayFastAPI, CORS, API key auth, request loggingPython, Uvicorn
Cognitive Router7-mode Ultra-Think, provider failover chainPython async
Agent EngineReAct loop, swarm orchestrator, 17 rolesPython asyncio
Memory System5-layer heat-scored recall, M0-M4PostgreSQL, pgvector
MCP Protocol13 tools, stdio/HTTP transportMCP SDK
ObservabilityCatalyst tracing, HALO analysis, PrometheusPython, Grafana
Security3-tier RBAC, Fernet encryption, Argon2idcryptography
PersistencePostgreSQL (TimescaleDB + pgvector), RedisDocker Compose

Data Flow

User Request → API Gateway (auth, logging)
    → Cognitive Router (select provider + reasoning mode)
    → LLM Provider (OpenAI / Anthropic / Ollama / llama.cpp / OpenRouter)
    → Agent Engine (ReAct loop, tool calls)
    → Memory System (recall + store)
    → Response (SSE stream or JSON)

Troubleshooting

Common Issues

Connection refused on port 8080

Another process is using the port. Kill it or change the port:

lsof -i :8080  # Find the process
kill -9 PID     # Kill it

# Or start on a different port
uvicorn main:app --port 8081

Ollama connection timeout

Ensure Ollama is running and accessible:

ollama serve &
curl http://localhost:11434/api/tags  # List models

Database connection failed

JEBAT runs in degraded mode without a database. To enable it:

# Set these in .env
JEBAT_DATABASE_ENABLED=true
DATABASE_URL=postgresql://jebat:password@localhost:5432/jebat_db

Import errors after restructuring

If you see ModuleNotFoundError, ensure the project root is on your Python path:

# From the project root
export PYTHONPATH="$PWD:$PYTHONPATH"
python -c "import jebat; print('OK')"
